User manual · part 1

What Dictionary Tasker is

Dictionary Tasker is a Clarion 12 IDE add-in that inspects the dictionary you currently have open in the editor. It started out doing three things SoftVelocity's shipped editor does not — browse the dictionary in structured views, batch-copy fields across many tables at once, and batch-copy keys across tables with automatic component re-wiring — and has since grown a tools catalogue covering lint, analysis, search, comparison, code generation, refactoring, visualization, and git integration.

It does not parse the .DCT binary on disk. It hooks into the live dictionary model that the editor is already holding in memory, reads from it, and writes back through the same APIs the editor itself uses.

Typical use case: you want every table in the dictionary to have guid, TimeStamp, ServerTimeStamp, and DeletedTimeStamp audit fields, plus a matching unique key on guid, without clicking through each table one at a time.

Beyond the three original features, the add-in has grown a tools catalogue — a grid of 20+ specialised tools spanning lint / validation, analysis, search, comparison, code generation, refactoring, and enterprise glue. Every implemented tool is shown in bold in the catalogue window; tools still marked (planned) would be placeholders, but as of this version every catalogue entry is live. See the Dictionary tools section below.

User manual · part 2

Installing and deploying

The add-in ships as a single DLL plus an .addin manifest, bytecode-compatible with Clarion 10, 11, 11.1, and 12 (every supported Clarion IDE loads .NET 4.0 add-ins from the same bin\Addins\Misc\ folder layout). Two ways to install:

Option 1 — installer (for end users)

install\build-installer.bat (requires Inno Setup 6) produces a single DictionaryTasker-Setup.exe under install\dist\ (version tracked in install\VERSION.txt, auto-bumped on each rebuild). The wizard shows a single page with four path entries (Clarion 12, Clarion 11.1, Clarion 11, Clarion 10) — fill in the ones you want, leave the rest blank. Each entry is pre-populated from auto-detection (C:\clarionXX, %ProgramFiles%\SoftVelocity\Clarion XX, or the (x86) variants). The wizard warns to close every targeted IDE, then copies the DLL + manifest into the right Addins\Misc\ClarionDctAddin\ subfolder for each filled-in version. Registers itself under Windows "Apps & features" for clean uninstall — the uninstaller sweeps every folder the installer wrote to.

Option 2 — deploy.bat (for quick local iteration)

  1. Close the Clarion IDE you're targeting. Clarion holds a lock on its add-in folder while running.
  2. Copy ClarionDctAddin.dll and ClarionDctAddin.addin into C:\clarionXX\bin\Addins\Misc\ClarionDctAddin\ where XX is 10, 11, 11.1, or 12. The repo ships a deploy.bat that does the Clarion 12 copy from the build output; duplicate it with the right path for any other version you're running.
  3. Launch the Clarion IDE. A new button with the Dictionary Tasker icon (book + magnifier) appears on the Standard toolbar. Clicking it is the sole entry point — the add-in does not add any main-menu entries. The button is disabled (rendered greyed-out) whenever no dictionary is open; it lights up the moment a .DCT tab is active.
Safety reminder: Batch operations mutate the open dictionary. The add-in always writes a timestamped .bak of the .DCT next to the file before any mutation, but you should still test new workflows on a throwaway copy of your dictionary first.

User manual · part 4

Browsing a dictionary

The browser dialog has three tabs, each a different view of the same model.

Tables tab

A flat ListView of every table with Name, Prefix, Driver, Fields count, Keys count and Description. Double-click a row to open the field browser for that table. The bottom button strip also lets you export the selected table or all tables to JSON.

Filter bar

Above the list:

  • Filter textbox — incremental substring match across every column (name / prefix / driver / fields / keys / description). Case-insensitive, re-runs on each keystroke.
  • Driver dropdown — populated from the distinct FileDriverName values actually used in the open dictionary ((all), MSSQL, TOPSPEED, MEMORY, …). Pick one to narrow the list to that driver.
  • Count label — "47 of 220 tables", updates as you filter.

Column sorting

Click any column header to sort by it; click again to flip direction. The active column shows a small ^ / v marker. Fields and Keys sort numerically (9 before 88, not lexicographically), everything else is case-insensitive string compare.

Right-click menu

Per-table actions, all scoped to whatever is selected:

  • Show fields... — opens the field browser.
  • Export to JSON... — routes through the JSON preview window.
  • Export SQL DDL... — preview window with a dialect picker.
  • Copy table name.
  • Lint this table... — per-table lint findings.
  • Fix fields... — editable Description / Picture grid with an auto-fill toolbar for descriptions (see the Validation & lint section).
  • Fix keys... — editable ExternalName grid with auto-fill for naming styles (UPPER_SNAKE, lower_snake, Pascal, camel, ...).
  • More dictionary tools... — opens the full tools catalogue.

Field browser

Opened from a Tables row, this modal shows every column on the selected table: Name, Full name (prefixed), Type, Size, Places, Dimensions, Picture, Initial, Heading, Prompt, Description, External name. The Inspect field button dumps every public property on the selected DDField — useful when a column displays blank and you want to confirm what Clarion actually holds.

Tree tab

A lazy-loaded TreeView: Dictionary → Tables → <TableName> → Fields / Keys / Relations / Triggers → individual items. Each leaf shows the item's readable scalar properties. Expanding a table with hundreds of fields is instant because children are only materialised when the parent is first expanded.

Relations tab

An interactive diagram of parent/child relationships between tables. Boxes are tables, arrows go from parent to child. Supports drag-to-move, Ctrl+wheel zoom, Fit-to-view, seventeen different auto-layouts (see the next section), and a Only Tables with Relations toggle that hides isolated tables.

User manual · part 6

Batch copy fields

Replicates selected fields from one source table into many target tables at once. Useful for audit columns, timestamps, soft-delete flags, GUID primary keys — anything you want to be present on every table without clicking through each one.

Workflow

  1. Pick the source table. The dropdown supports incremental search — type anywhere in a table name.
  2. Check the fields you want to copy. Only fields that exist in the source table are listed.
  3. Check the target tables. The source table is hidden from the target list. Use Select all / Clear all for quick batch selection.
  4. Pick a conflict mode:
    • Skip that field (recommended) — fields that already exist in a target by label are silently skipped.
    • Abort on first conflict — the whole batch is cancelled at the preview stage if any conflict exists.
  5. Click Preview and apply…. A modal lists every planned add or skip row. Click Apply changes to commit.
  6. A progress dialog shows the current target.field as it processes, with a Cancel button and progress bar.
  7. Dictionary is marked dirty. Press Ctrl+S in Clarion to persist, close the dictionary tab and reopen it to refresh the editor's view.
Backup: before any write, the add-in copies your mydict.dct to mydict.dct.tasker-bak-<yyyymmdd-hhmmss> in the same folder. If the copy fails the batch aborts.
User manual · part 7

Batch copy keys

Copies keys from a source table to multiple targets. Unlike fields, a key has components that reference specific fields in its own table — those references have to be re-wired to the target table's matching fields when the key lands.

How re-mapping works

  1. Each source-key component is identified by the Label of the field it points at.
  2. When copying to a target, the add-in looks for a field with the same label in the target. If found, the new key's component is re-wired to point at that target field.
  3. If any required field is missing from the target, the whole key is skipped (or the whole batch aborted, depending on the conflict mode).

Auto-generated external names

A key's ExternalName normally collides when copied verbatim (two keys with the same external name break the save). The add-in rewrites each copy's ExternalName to <TargetTableName>_<KeyLabel> so every target gets a unique, predictable value. E.g. key guidkey copied into BITACORA gets BITACORA_guidkey.

Run field copies first. If a key references guid and TimeStamp, the targets must already have those fields (same label) before the key copy can land. Batch-copy the fields first, then batch-copy the keys.

Output schema

The hand-picked schema is stable and machine-parsable. Tables get a curated scalar list (name / prefix / driver / flags / …) plus arrays for Fields, Keys, Relations, Triggers, Aliases. Fields go through a second hand-picked scalar list (type / size / picture / heading / prompt / external name / booleans). Keys / Relations / Triggers / Aliases are auto-dumped by reflection but with a cycle guard and depth / size caps.

}

Safety rails

The reflection-based auto-dumping (for keys / relations / triggers / aliases) has caused IDE-level crashes on wide tables where a SoftVelocity property exposed a back-reference collection that looped. The generator now has:

  • Cycle guard — a reference-equality HashSet<object> tracks visited instances for the whole run; repeat visits emit "<cycle: TypeName>" and stop recursing.
  • Depth cap — reflection recursion stops at depth 3.
  • Size caps — the generator aborts after 200 000 nodes or ~50 MB of text, writing a trailing // NOTE: output truncated comment.
  • Per-collection cap — 10 000 items per enumerable. Defensive try/catch around the enumerator itself in case a property throws mid-walk.
  • UI catch — the Export Selected / Export All handlers wrap the call in a try/catch; an OutOfMemoryException (or anything else) surfaces as a modal error dialog instead of killing Clarion 12.

Under the hood

JsonExporter.cs emits the pretty-printed output; JsonFormatter.cs re-formats between Pretty / Tabs / Minified as a string-aware state machine; JsonParser.cs is the recursive-descent parser used by Tree and the four structured views; JsonViewTransforms.cs hosts the Table / YAML / Path list / Schema renderers. The whole stack is dependency-free — no Newtonsoft, no System.Web.

User manual · part 9

View data (TPS & SQL)

Right-click any table in the Tables list and pick View data... to see the first N rows of the table's actual contents in a read-only grid. Useful for sanity-checking schema changes, spot-checking a migration, or just peeking at live data without leaving the IDE.

Three paths, one dialog

The dialog adapts to whatever driver the table uses. Clarion dictionaries mix physical-file drivers (TOPSPEED, DBASE, …) with network drivers (MSSQL, ODBC, …); each needs a different reader.

DriverHow we read itStatus
MSSQL Direct ADO.NET via System.Data.SqlClient. Connection string resolved from the table's OWNER('…') argument, saved per-dict, re-promptable via Change connection…. Full support, including types.
TOPSPEED (.TPS) Direct managed parse of the TPS bytes via the bundled tps-parse-net library (Apache-2; see Credits). Does not use Clarion's ClaTPS.dll or the runtime's CFile pipeline — the add-in reads the file format itself, so it works in-process without dragging in native driver state. Strings, dates, times, integers, decimals, BCD, floats, doubles, fixed / pascal / zero-terminated strings, groups, arrays, memos (placeholder render). Edge-case TPS files (encrypted, unusual schema variants) may fail to parse — the parser is community reverse-engineered.
All other file drivers Falls through to ClarionFileAccessor, which drives the Clarion .NET runtime via reflection. Currently gated to a graceful error for non-TPS; this is the hook for future drivers (ASCII, DOS, DBASE3/4) if we ever need them. Not yet implemented — surfaces a clear error.

Backup viewer — embedded TopScan

Every View data dialog also carries Open in TopScan and Embed TopScan buttons. TopScan is SoftVelocity's own TPS viewer, shipped with Clarion. Open in TopScan launches it as a separate window; Embed TopScan uses Win32 SetParent to reparent its top-level window into a panel inside the View data dialog — no second taskbar entry, no alt-tab to switch between the grid and the hex view.

TopScan is always available as a fallback: it uses Clarion's own private TPS codec, so any TPS file it can open is guaranteed to render, including exotic encrypted / versioned variants our own parser doesn't know about.

Why bundle a TPS parser — why not use Clarion's runtime?

Earlier drafts of View data tried to drive Clarion's own .NET runtime: SoftVelocity.Clarion.FileIO.Clarion.CFile via an emitted subclass + SetupFile(xmlDoc, 0) with a synthesised schema. That pipeline works all the way through file open and record iteration — the native ClarionDrv.dll loads, the TPS driver accepts the Open, Next() returns 0 (success) — but Bytes() and Records() both return 0 and the record buffer stays empty.

The reason: Clarion's native runtime calls back into managed code via CLegacy.GetValue to ask for record-layout details. Clarion-generated CFile subclasses satisfy that callback contract through their __CLA_Store / __CLA_Retrieve / __CLA_GetRecord overrides, which marshal a CGroupBuffer against a CGroup describing each field. Our runtime-injected subclass stubs those abstracts as no-ops, so the native side silently skips record marshaling. Reimplementing that callback contract would effectively mean reimplementing Clarion's code generator — way outside the scope of a read-only data viewer.

Parsing TPS directly bypasses the problem. Erik Hooijmeijer's open-source reverse-engineering of the TPS format (and Adam Burger's C# port of it) give us a pure managed reader that reads the bytes off disk and hands us back strongly-typed field values. No native DLLs, no callback contracts, no IDE crash risk.

Under the hood

ViewDataDialog.cs hosts the grid and the toolbar. For TPS drivers, TpsDirectReader.cs wraps the bundled parser in TpsParse/ and emits a ClarionFileAccessor.ReadResult (shared shape with the SQL path so the grid stays driver-agnostic). DCT field labels map to TPS field names by case-insensitive match on the suffix after the PRE: prefix — Clarion's internal naming convention — with a small loss for audit-only DCT fields that don't exist physically in the TPS.

One local patch to the bundled parser: DataRecord.ParseValues now seeks to each field's declared Offset before reading, rather than advancing the stream position sequentially. Without that, OVER()-style aliased fields (common in Clarion — multiple named fields sharing the same bytes as a parent group) come out as mid-string garbage. One-line change, documented in the patched source.

Dictionary tools · category 1

Validation & lint

Read-only scanners that report structural or stylistic issues. None of these mutate the dictionary.

Lint report

Full-dictionary scan covering structural AND style issues:

  • Tables: empty, no keys, no primary key, no relations.
  • Keys: empty components, duplicate component sets, empty ExternalName.
  • Duplicate ExternalName across the dict — two keys sharing an ExternalName break SQL drivers (most RDBMSs forbid two indexes of the same name in one schema). Emits one Error finding per occurrence so both sides of every collision show up, each with the other owners listed in the Message. Single-table scans also surface cross-table collisions involving the scanned table.
  • Illegal ExternalName shape — driver-aware checks on every table's FullPathName and every field's / key's ExternalName:
    • : anywhere → Error (illegal-external-name; Clarion field-prefix leak, breaks MSSQL / Postgres / MySQL / MariaDB).
    • Whitespace → Error (illegal-external-name).
    • Starts with a digit → Warning (illegal-external-name).
    • Characters outside [A-Za-z0-9_$#.] → Warning (unusual-external-name-chars).
    • Length over driver limit → Warning (long-external-name). Limits: Postgres 63, MySQL / MariaDB 64, MSSQL / SQLite 128.
  • Fields: no description, no picture, whitespace in label.
  • Picture shape — DATE needs @d*, TIME @t*, numerics @n*, STRING forbids @d/@t/@n. LONG / ULONG accept @n, @d, or @t because Clarion uses LONG as the underlying storage for dates (days since epoch) and times (centiseconds since midnight).
  • Money-ish labels (amount, precio, …) on numerics without @n$ → info-level suggestion to use a currency marker.

Findings grouped by severity (Error / Warning / Info) with a filter box for quick narrowing. Also available as a per-table scan via right-click → Lint this table on the Tables list, Tree view, or Relations diagram.

Fix fields — editable grid

The repairable subset of the lint (missing description, missing picture, malformed picture) typically accounts for the bulk of findings. Instead of clicking into each field one at a time, Fix fields... opens a DataGridView with one row per flagged field and the Description + Picture columns editable inline:

  • Live re-check — edit a DATE field's picture to @d6 and the Issues cell clears on the spot; type a bad value and the message reappears. Dirty cells get a soft amber tint so you can see at a glance what will be written.
  • Apply changes... confirms the counts (description vs. picture), writes a .tasker-bak-<timestamp> backup of the .DCT, then pushes each edited row through FieldMutator.SetStringProp — the same reflection path Safe rename and Batch rename use. Failures are reported per-field.
  • Rescan picks up anything added to the dict in the meantime.

Auto-fill for Description

Toolbar above the grid with a strategy dropdown plus Fill blanks / Fill all (overwrite) buttons. Strategies:

StrategyExample
Best available (default) heading → prompt → humanized label, first non-blank.
Humanized label first_name → "First name" · CreatedOn → "Created on" · NO_DE_RECUPS → "No de recups" · IDField → "Id field". Splits on _/-/. plus camelCase boundaries (including the trailing-abbreviation case where ID precedes Field).
Column headingCopy DDField.ColumnHeading verbatim.
Prompt textCopy DDField.PromptText verbatim.
Label verbatimCREATED_BY stays CREATED_BY.

Suggested values land in the editable Description cell (amber tint, Issues column re-evaluates) but nothing persists until you click Apply changes. Picture values stay manual — auto-generating a picture from data type alone is too lossy.

Launched from the Fix fields... button on the Lint report dialog, or from right-click → Fix fields... on any table list (Tables tab / Tree / Relations). Scope follows the launcher: whole-dict scope from the lint report, single-table scope from a right-click.

Fix keys — editable grid

Same shape as Fix fields but targets DDKey. Flags keys with:

  • empty ExternalName,
  • zero components,
  • a component signature that duplicates another key in the same table,
  • an ExternalName that duplicates one elsewhere in the dict,
  • an ExternalName with illegal characters (the same driver-aware rules the lint report uses — :, whitespace, leading digit, unusual chars, over length limit).

Only the ExternalName column is editable inline; the rest is context (table, key, type, components, Unique / Primary flags, issues). Summary bar reports blank: N · duplicated: M · illegal: K · E edited · R still have outstanding issues.

Show filter

The Show: dropdown narrows the grid without changing the underlying data — switching views is instant.

  • All issues (default)
  • Blank ExternalName only
  • Duplicated ExternalName only
  • Illegal characters / length only

Auto-fill for ExternalName

Three dropdowns (all persisted across sessions) plus Fill blanks / Fill all (overwrite):

DropdownOptions
Auto-fill style UPPER_SNAKE · lower_snake · Camel_Snake · Pascal · camel · idx_snake · Key only (no owner) · Owner prefix + key as-is
Owner Table name · Prefix (strips trailing : on the Clarion prefix)
Key Label only (after the :) · Full key name. Default is Label only because SQL forbids : anyway.

That's 8 × 2 × 2 = 32 combos. Common picks on a freshly-imported Clarion dict: UPPER_SNAKE + Table name + Label only yields CLIENTES_GUIDKEY; UPPER_SNAKE + Prefix + Label only yields CLI_GUIDKEY. Fill all (overwrite) iterates the full row set — so rows currently filtered out of the grid by the Show dropdown still get regenerated.

Typical workflow for a brand-new MSSQL target: Lint report → see illegal-external-name / duplicate-external-name errors → Fix keys... → pick UPPER_SNAKE + Table name + Label only → Fill all (overwrite) → Apply → every index gets a unique, legal TABLE_KEY name in one shot.

Launched from the Fix keys... button on the Lint report dialog, or right-click → Fix keys... on any table list.

Picture consistency (standalone)

Same per-type rules as the lint report, plus a cross-table consistency pass: when the same field label appears on many tables with divergent (type, picture) combos it flags the label as inconsistent. That cross-table pass needs whole-dict scope so it doesn't live on the per-table right-click.

Naming conventions

Configurable rules with checkboxes — the scan re-runs whenever you toggle a rule:

  • Tables UPPERCASE.
  • Prefixes 2–4 uppercase characters.
  • Field labels contain no whitespace.
  • Field labels do not start with a digit.
  • Key names include a convention marker (:PK, :BY, or KEY).

Each violation is shown alongside a suggested fix (e.g. the whitespace rule emits oldLabel → old_Label).

Dictionary tools · category 2

Analysis & stats

Health dashboard

Four-panel read-only summary of the dictionary:

  • Top bar: totals — tables, fields, keys, relations, average fields per table.
  • Top-left: top-10 largest tables by field count (ListView).
  • Top-right: driver distribution bar chart (MSSQL vs TopSpeed vs …).
  • Bottom: histogram of relations-per-table bucketed 0 / 1 / 2 / 3 / 4 / 5 / 6-9 / 10+.

Dead tables

Lists tables with zero relations and (optionally) zero references from any other table's keys / relations / triggers. Good candidates for deletion. Read-only — just the report.

Duplicate fields

Groups every field by the triple (label, type, size) and shows every group with two or more instances. Each row: label, type, size, count, comma-separated list of tables. Useful for spotting candidates for extraction into a shared audit-field group.

Dictionary tools · category 3

Search & navigation

Global search

Debounced full-dictionary search. Type a term; matches appear in a ListView with Kind / Table / Item / Context columns. Tick which kinds to search (Tables / Fields / Keys / Relations / Triggers / Descriptions) and whether to treat the term as regex or plain substring.

Descriptions include table / field Description, Prompt, Heading, and trigger bodies.

Where used

Pick a table + a field. The tool scans every table's keys, relations, and trigger bodies for references to that field:

  • Key components on the home table (match via field-object reference).
  • Relation components anywhere in the dict that point at the field.
  • Trigger bodies that mention the field by LABEL or PREFIX:LABEL with a word-boundary check to avoid partial matches inside other identifiers.

Path finder

Answers the question "how is CLIENTES related to INVOICES?" Picks two tables; runs breadth-first search on the undirected relation graph; shows every hop as TABLE_A --(relation_name)--> TABLE_B. "No relation path" is a valid answer — the two tables are in disconnected components.

Dictionary tools · category 4

Compare & diff

Compare tables

Refactoring (mutation)

Every tool in this section mutates the open dictionary. A timestamped .tasker-bak-<yyyymmdd-hhmmss> copy of the .DCT is written before any change; if the backup fails the batch aborts. The shared reflection plumbing lives in FieldMutator.cs.

Safe rename field

Rename one field's label across all downstream references.

  • Keys and relations update automatically. They hold references to the field object, not the label string — setting the new label on the field instance is enough.
  • Trigger bodies are patched by text replace (optional). The regex \b(PREFIX:LABEL|LABEL)\b catches both the prefixed and bare forms, with word-boundary anchors so partial substrings inside other identifiers are not touched. Uncheck the option if you'd rather update triggers manually.

The preview lists every affected row (field itself, each key / relation reference, each trigger body occurrence) before you confirm.

Regex find/replace across selected text properties of every field in the dict, optionally scoped by a table-name regex. Checkboxes pick which properties to touch: Label / Description / Heading / Prompt. The replacement supports standard $1 backreferences. Preview the full plan before applying.

Pattern-match field labels with a regex, then set a new DataType, FieldSize, and / or ScreenPicture. Any of the three target values can be blank, meaning "preserve the current value". Preview lists before → after per affected field.

Canonical propagation of the audit field pack (Guid + CreatedOn + CreatedBy + ModifiedOn + ModifiedBy + DeletedOn) onto every selected target table.

  3. Tick target tables in the second list (All / None helpers provided).

Dictionary tools · appendix

Snapshot format (*.tasker-snap)

The Compare dictionaries and Change-log generator tools read and write a small tab-delimited text format optimised for simplicity and diffability. It captures the structural shape of the dictionary — tables, fields, keys, relations — but not data, triggers' bodies, or editor-specific metadata.

Line format

First column is a section tag; remaining columns are the record fields, tab-separated. Embedded tabs, newlines, and backslashes are escaped (\\, \t, \n, \r).

SNAP  <ver>  <dict-name>     <file-name>  <captured-utc-iso>
T     <name>  <prefix>   <driver>    <description>
F     <label> <type>     <size>      <picture>     <description>
K     <name>  <type>     <unique0/1> <primary0/1> <comma,sep,components>
R     <name>  <related-table>

T opens a new table block; subsequent F / K / R lines belong to it until the next T. The format is deliberately line-oriented so it diffs cleanly in git and stays readable to a human without a special viewer.

Implementation lives in DictSnapshot.cs. The diff engine that compares two snapshots is DictDiff.cs; its RenderMarkdown(before, after, diff) is what both Compare dictionaries and Change-log generator feed into their Markdown export.

Developer manual · part 4

Mutation patterns (writing to the live dictionary)

Simply adding a DDField to DDFile.Fields does not make it stick. The Clarion editor keeps a native TPS record store (Win32Dictionary) alongside the managed model, and the save pipeline iterates that native store. Field/key state has to be configured before registration for the internal canonical methods to actually persist the record.

Field copy recipe (from FieldCopier.CopyAndAdd)

  1. sourceField.Copy(parent, quiet=false) — clone with a new Guid, parent wired.
  2. Reflection-set backing fields on the copy before registration: parentItem, storageLocation=Application, stored=true, itemHasChanged=true, recordOrder.
  3. Try the internal DDFile.InsertField(DDField) — this is the lifecycle path that allocates a _C6Ident and creates the native record.
  4. If not yet registered, try DDField.Insert(position) on the instance itself.
  5. As a last resort, call the collection's own Add(DDField) on the managed list.

Key copy recipe (from KeyCopier.CopyAndAdd)

Similar shape but with two critical differences:

  • Call GenerateNewId() right after Copy. Without it, the new key's GUID collides with its source.
  • Do not set stored=true. For keys, that flag routes save through DoUpdate, which looks up an existing on-disk record by GUID and throws "Record X not found." Leaving it false (the value Copy produces) routes save through DoInsert instead.
  • Assign a fresh Order = current Keys count, otherwise the native KEYS-BYORDER index throws "Creates Duplicate Key".
  • Re-wire KeyComponents: for each source component, find the target field with the same Label and point the new component's Field / FieldId there.

Property edits (rename / retype / Fix fields) — FieldMutator

The refactor tools and the Fix-fields grid don't add or remove fields — they change existing properties on an already-registered field instance. Sounds simpler than the insert recipe, but it has its own subtle failure mode: the managed model takes the new value happily (so the IDE shows it immediately), while the save pipeline silently drops it because the native TPS change tracker never learned that THIS field was modified. FieldMutator.SetStringProp therefore fires a fan-out of notifications after each write:

  1. Back up the .DCT first via FieldMutator.Backup.
  2. Write the value — try the public setter first; if the property is read-only or the setter discards the value, fall back to the non-public backing field walking the class hierarchy (name, _name, _Name, m_Name, …).
  3. Flip per-field flags: itemHasChanged = true, stored = true, Touched = true.
  4. Invoke every per-field "I've changed" method we know about by name: SetInFile, Touch, DoUpdate, NotifyUpdate, NotifyChanged, MarkChanged.
  5. Walk up to the parent DDFile and call ChildListTouched, FieldChanged(field), ChildTouched(field).
  6. Find the owning collection — FindOwningCollection walks Fields / Keys / Relations / Triggers / Aliases / Components on the parent and picks the one that contains the target by reference. That's how the same SetStringProp path works for fields and keys without caller-side special casing — a DDKey edit naturally registers on parent.Keys, not parent.Fields.
  7. Register in the change-tracker dictionary on that owning collection — candidate names modifiedItems, changedItems, updatedItems, dirtyItems, addedItems — keyed by the item's Id. This is the modification-side analogue of the addedItems hook FieldCopier uses for inserts, and in practice it's the one the save pipeline reads when deciding which records to write back to the native TPS store. Without it, the edit stays in the managed model and is lost when the .DCT saves.
  8. Call ChildListTouched on the DataDictionary.
  9. Run FieldMutator.ForceMarkDirty(dict, viewContent, result) for the usual IsDirty / OnIsDirtyChanged cascade so Clarion's Save button lights up.

Every probe that fired writes a short token into the result log (setter / item=chg / stored / coll.modifiedItems / file.ChildTouched …) so if a future Clarion build routes through a member name we haven't listed, the per-field log tells us exactly which probe to add.

Marking dirty for save

The editor's Save button is driven by SharpDevelop's ICanBeDirty.IsDirty on the DataDictionaryViewContent, not on the model. After a successful batch, we set:

// Model side
TrySetProp(dict, "IsDirty", true);
TryInvokeNoArgs(dict, "DoIsDirtyChanged", nonPublic: true);
TryInvokeNoArgs(dict, "ChildListTouched", nonPublic: true);

// Editor side — this is what actually enables the Save menu
TrySetProp(viewContent, "IsDirty", true);
TryInvokeNoArgs(viewContent, "OnIsDirtyChanged", nonPublic: true);
Developer manual · part 5

Custom toolbar icon

This was surprisingly hard. SharpDevelop resolves the icon="…" attribute on <ToolbarItem> at toolbar-construction time, which runs before any /Workspace/Autostart class. Registering a custom bitmap in ResourceService from autostart is too late — the button already has its Image.

Clarion's toolbar items are a custom class ToolBarCommand (a ToolStripItem subclass) that paints its icon from its Codon. Setting item.Image does nothing; reflection-setting every Image-typed field on the item is also a no-op because the paint path doesn't read those fields — it reads from the codon. Even hooking ToolStrip.Paint to paint-over only fires for full-strip repaints, so hover state restores the stock icon.

The working fix: replace the ToolStrip renderer

Subclass ToolStripProfessionalRenderer, override OnRenderItemImage to draw our bitmap for our button and skip base, and swap the instance onto the owning ToolStrip:

internal class DictTaskerToolbarRenderer : ToolStripProfessionalRenderer
{
    readonly Image icon;
    public DictTaskerToolbarRenderer(Image icon) { this.icon = icon; }

    protected override void OnRenderItemImage(ToolStripItemImageRenderEventArgs e)
    {
        if (icon != null && StartupCommand.IsOurToolbarItem(e.Item))
        {
            var r = e.ImageRectangle;
            if (r.Width > 0 && r.Height > 0)
                e.Graphics.DrawImage(icon, r);
            return; // skip base — stock icon is never drawn
        }
        base.OnRenderItemImage(e);
    }
}

The manifest keeps referencing a known-good icon like Icons.24x24.StartPage so the toolbar always builds. StartupCommand polls on a Timer until our button is found, then swaps the renderer once per owning ToolStrip. Every other item keeps default styling because the subclass delegates to base for non-matching items.

Disabled-state rendering

The custom renderer inspects e.Item.Enabled inside OnRenderItemImage. When the item is disabled (see the conditional-enable section below), the renderer applies an ImageAttributes with a luminance-weighted ColorMatrix (0.30R + 0.59G + 0.11B) and alpha dropped to 0.45 — producing a desaturated, semi-transparent version of the icon at paint time. No separate greyed asset ships in the DLL.

Conditional enable — "dictionary open"

The toolbar button is disabled while no .DCT is loaded in the IDE. SharpDevelop supports this declaratively via IConditionEvaluator: register a named condition and wrap the ToolbarItem in a <Condition> element. The framework re-evaluates the condition on workbench state changes (view opened, closed, switched) and sets the item's Enabled automatically.

The evaluator class

public class DictionaryOpenCondition : IConditionEvaluator
{
    const string DictViewFullName =
        "SoftVelocity.DataDictionary.Editor.DataDictionaryViewContent";

    public bool IsValid(object caller, Condition condition)
    {
        var wb = WorkbenchSingleton.Workbench;
        if (wb == null) return false;
        foreach (var vc in (IEnumerable)wb.ViewContentCollection)
            if (vc != null && vc.GetType().FullName == DictViewFullName)
                return true;
        return false;
    }
}

Relations diagram internals

The diagram is one Panel (RelationsDiagramPanel) with custom OnPaint. Everything is drawn via System.Drawing — no external graphics library.

State

  • nodes — one TableNode per table (box position + hover state).
  • edges — one RelEdge per relation (source node, target node, label).
  • zoom — applied via Graphics.ScaleTransform during paint.
  • currentEdgeStyle — Default / Orthogonal / Arc / Chord. Each layout opts into its style.

Reflection deep dive · chapter 1

What reflection is

Reflection is .NET's facility to inspect and use types by name at runtime instead of referencing them at compile time. Every object carries a Type handle via GetType(); from that handle you can enumerate members, look one up by string, read values, write values, and invoke methods — all without the compiler ever having seen the type.

The normal (compile-time) way

using SoftVelocity.DataDictionary;

var dict   = (DDDataDictionary)someObj;
var tables = dict.Tables;
foreach (var t in tables)
    Console.WriteLine(t.Label);

Requires DataDictionary.dll as a project reference. The compiler verifies that Tables and Label exist with those exact names and types.

The reflection way

var tables = someObj.GetType().GetProperty("Tables").GetValue(someObj, null);
foreach (var t in (IEnumerable)tables)
    Console.WriteLine(t.GetType().GetProperty("Label").GetValue(t, null));

The compiler knows nothing about SoftVelocity types. "Tables" and "Label" are string keys resolved by the runtime when the code executes.

Reflection deep dive · chapter 2

Why we use reflection here

  1. No hard SoftVelocity DLL reference. The add-in project only references ICSharpCode.Core and ICSharpCode.SharpDevelop — the add-in framework itself. It never references DataDictionary.dll, DataDictionaryEditor.dll, or any Clarion.* assembly. That means the add-in builds identically on any machine with a .NET Framework 4.0 targeting pack, even without Clarion installed.
  2. Portability across Clarion point-releases. If SoftVelocity ships 12.0.14005 with rev'd assembly versions or renamed internal classes, a compile-time-bound add-in would either fail to load (strong-name mismatch) or crash the first time it touched a vanished type. Reflection-based lookup by string name degrades gracefully — a missing member just returns null.
  3. Access to non-public members. Several members we must set to make persistence work — parentItem, stored, itemHasChanged, addedItems, InsertField, InsertKey, GenerateNewId — are internal, private, or protected. A compile-time reference can't call them. BindingFlags.NonPublic | BindingFlags.Instance bypasses visibility.
  4. Defensive probing. When we didn't know whether key components lived on Components, KeyComponents, Fields, Segments, or something else, the code tried each candidate name in turn and used whichever returned a collection. Compile-time code can't do that.
  5. Patching third-party controls. The toolbar-icon fix walks every ToolStrip on every open Form, finds our button by its tooltip text, and replaces the owning toolbar's Renderer. We never had that control under our ownership; reflection and WinForms control-tree walking let us reach in and take over one specific render path.
Reflection deep dive · chapter 3

The patterns we use

Look up a public property, returning null on failure

From DictModel.cs. Used hundreds of times; the foundation of everything else.

public static object GetProp(object o, string name)
{
    if (o == null) return null;
    var p = o.GetType().GetProperty(name, BindingFlags.Public | BindingFlags.Instance);
    if (p == null || !p.CanRead || p.GetIndexParameters().Length > 0) return null;
    try { return p.GetValue(o, null); } catch { return null; }
}

Invoke an internal method with a typed parameter

From FieldCopier.cs. This is the payload of the fields batch copy.

var insertField = targetTable.GetType().GetMethod(
    "InsertField",
    BindingFlags.NonPublic | BindingFlags.Instance,
    null,
    new[] { newField.GetType() },
    null);

if (insertField != null)
{
    try { insertField.Invoke(targetTable, new object[] { newField }); }
    catch (TargetInvocationException tie) { throw tie.InnerException ?? tie; }
}

Note the GetMethod overload that takes a Type[]. It's the only way to disambiguate overloaded methods precisely. TargetInvocationException wraps the real exception from the invoked method; we unwrap with InnerException so the caller sees the actual failure.

Walk base types to set an inherited non-public field

Type.GetField with BindingFlags.NonPublic does not return inherited non-public fields. You have to walk the class hierarchy yourself: 

static bool TrySetObjectField(object target, string name, object value)
{
    if (target == null) return false;
    var t = target.GetType();
    while (t != null && t != typeof(object))
    {
        var f = t.GetField(name,
            BindingFlags.Public | BindingFlags.NonPublic
          | BindingFlags.Instance | BindingFlags.DeclaredOnly);
        if (f != null)
        {
            try { f.SetValue(target, value); return true; }
            catch { return false; }
        }
        t = t.BaseType;
    }
    return false;
}

BindingFlags.DeclaredOnly restricts each call to fields declared directly on the current type, and the loop takes care of the inheritance chain. This is how we get access to parentItem on DataDictionaryItem through a DDField instance.

Defensive name probing

We rarely know the canonical name of an internal field. Instead, we try a list and use whichever resolves. From KeyCopier.FindComponents: 

static IEnumerable FindComponents(object key)
{
    string[] names = { "Components", "KeyComponents", "Fields", "KeyFields",
                         "Segments", "Parts", "Children", "Items", "FieldList" };
    foreach (var n in names)
    {
        var v = DictModel.GetProp(key, n) as IEnumerable;
        if (v != null && !(v is string)) return v;
    }
    // Last-resort fallback: scan every public IEnumerable property for one
    // whose items carry a Field reference.
    // ...
    return null;
}

Trade-offs

Trade-offWhat it costs usHow we mitigate it
No compile-time checking Typo "Tabels" → runtime null, not a build error. Defensive null-checking. Log lookups to %TEMP% so discovery of missing members is fast. Unit-style smoke tests during bring-up (the Reflection dump and Discover write API debug menu entries).
No IntelliSense Can't autocomplete SoftVelocity member names in the IDE. Inline tree-view browser lets the dev list every scalar property on a live instance. It's slower than IntelliSense but works on any build of Clarion.
Slower than direct calls Reflection invocation is 10–100× slower than a static method call. Negligible at our call volumes (hundreds of invocations per batch, not millions). No optimisation needed.
Brittle to renames A future SoftVelocity release renaming parentItem silently breaks copy. Defensive candidate-name probing. Fallback chains (try internal method, else public method, else collection Add). Every batch run logs which path succeeded so regressions surface quickly.
Visibility rules bypassed We're deliberately reaching into API surfaces SoftVelocity did not mean to expose. Backup-before-write is enforced for every batch. Every mutation goes through Clarion's own lifecycle methods (Copy, InsertField) where possible, not by hand-crafting records in the native TPS store.

The net effect: we trade compile-time safety for portability + access to non-public members, which was essential because the entire value proposition of this add-in is write back to a live dictionary that was never designed for third-party mutation.
